# Threat Model: tmi-ux (TMI threat-modeling web front-end)

## 1. System context

tmi-ux is the Angular 21 / TypeScript single-page web front-end for **TMI**, a collaborative threat-modeling application (public instance at tmi.dev). It lets security engineers author threat models, data-flow diagrams (DFDs), threats, assets, and markdown/Mermaid notes; edit diagrams collaboratively in real time over WebSockets; and run intake/triage flows. RBAC roles are reader / writer / owner at the resource level plus two global roles, administrator and security-reviewer, all derived from JWT claims.

The application is a **thin client**. The authoritative back-end is a _separate_ Go server (github.com/ericfitz/tmi), reached over REST (OpenAPI 3.0) and WebSockets; it is **not in this repo**. This repo's only server-side component is a thin Node/Express 5 `server.js` (~154 lines) that statically hosts the built Angular bundle, serves a runtime `GET /config.json` derived from `TMI_*` environment variables, and optionally reverse-proxies `/api/*` (including WebSocket upgrades) to a configured back-end. It performs no authentication, sets no cookies, and runs no session middleware. The repo also ships Python i18n tooling under `scripts/` (developer-only, not a runtime web surface).

The code base is security-conscious, with a visible remediation audit trail: a named AUTH-VULN / CodeQL campaign, two in-tree security analysis documents (`docs/AUTHZ_ANALYSIS.md` and `docs/SESSION_MANAGEMENT_ANALYSIS.md`, both 2026-03-05), and extensive `pnpm.overrides` hardening the transitive dependency tree. **Important reconciliation:** those two analysis documents describe an _older_ auth design (access + refresh JWTs in localStorage, AES-256-GCM encrypted with a browser-fingerprint-derived key). The current code has since **migrated to HttpOnly-cookie session management** (commit a7d070ce): tokens are issued by the back-end, held in HttpOnly cookies, and are never readable by JavaScript; the SPA tracks only session timing in memory, and the fingerprint key-derivation scheme was replaced (db1484e8) by a non-extractable IndexedDB CryptoKey. This model scores against the **current** HttpOnly-cookie code and treats the doc findings as historical evidence that the auth/session surface is fertile, not as live token-exfiltration vulnerabilities. Because the heavy authorization and data-storage enforcement lives in the out-of-repo Go back-end, several front-end threats are **contingent on back-end behavior** the code in this repo cannot confirm (see Open questions).

## 2. Assets

| asset | description | sensitivity |
|---|---|---|
| TMI session (HttpOnly cookie) | Primary auth credential, backend-issued, held in HttpOnly cookies never readable by JS; SPA tracks only session timing in-memory. Compromise = account takeover. | critical |
| Threat-model documents | Core product data: threats, assets, diagrams, notes, ACLs, `is_confidential` flag. They describe an org's real security weaknesses; disclosure is high-impact. Persisted server-side. | critical |
| TMI backend API | Authoritative store for all data, auth, and collaboration behind `environment.apiUrl`. Availability/integrity gate the whole product. Out of this repo. | critical |
| OAuth state / PKCE verifier | OAuth `state` in localStorage, PKCE `code_verifier` in sessionStorage; protect the OAuth callback from CSRF / code-injection. | high |
| Delegated content-provider tokens | Server-minted OAuth tokens (Google Drive / Microsoft), short-lived picker tokens, and a browser-minted GIS `access_token` granting direct access to the user's external Drive. | high |
| DFD diagrams / cells | Data-flow diagrams the user edits, part of each threat model; architectural detail an attacker would want. Rendered by @antv/x6. | high |
| Authorization / ACL data | Per-threat-model `Authorization[]` (reader/writer/owner principals), `owner`, `security_reviewer`. Tampering = privilege escalation over a model. | high |
| Real-time collaboration session | Live multi-user diagram-editing session over WebSocket; carries TM/diagram IDs, participants, permissions. Hijack exposes/edits a live model. | high |
| SPA static server (Node/Express `server.js`) | Serves the app, `/config.json`, and the optional API proxy. Its availability determines whether anyone can load the app. | high |
| User identity / profile | `UserProfile`: provider, email, display_name, group memberships, `is_admin` / `is_security_reviewer`. PII plus privilege data, in-memory. | medium |
| Runtime config (`/config.json`) | Env-derived client config (apiUrl, auth defaults, security flags). Not secret, but controls client security posture and the API endpoint. | medium |
| User preferences cache | localStorage UI prefs / theme / language. Non-sensitive but a tampering / XSS sink. No auth tokens stored here. | low |

## 5. Deprioritized

| threat | reason |
|---|---|
| Memory-safety / RCE in native parsers (buffer overflow, UAF) | Out of scope: target is an Angular/TypeScript/Node web app with no C/C++; no native memory-safety surface. |
| SQL/NoSQL injection against the application database | Asset not present in this repo: all persistence is in the out-of-repo Go backend; the SPA constructs no database queries. |
| Server-side template injection / SSRF from user input | `server.js` performs no templating and makes no user-controlled outbound requests; the only proxy target is operator-set (covered by T14). |
| Token exfiltration from browser storage via XSS | Mitigated by design: post-migration (a7d070ce) tokens live in HttpOnly cookies unreadable by JS; the historical localStorage/fingerprint-key exposure (SESSION_MANAGEMENT_ANALYSIS ARCH-1) no longer applies. XSS session-riding is captured by T4 instead. |
| JWT in WebSocket URL leak (CWE-598) | Mitigated: token stripped from WS URL (91aa3dc8) and ticket/session-bound WS auth adopted (49590bd7); the historical AUTH-001 finding is closed. |
| Python i18n tooling command/code injection | Not a runtime web surface: `scripts/` run only at developer/dev time; subprocess calls are list-form (no `shell=True`) and no eval/exec/pickle/yaml.load is present. |
| Secrets baked into the client bundle | Asset not present: no client_secret/client_id/API keys in `src/environments`; OAuth uses PKCE with backend-driven provider discovery; no source maps shipped to prod. |

## 6. Open questions

These could not be answered from the front-end code alone and should be confirmed with a TMI back-end owner (they gate the likelihood of several "contingent" threats above):

- **Back-end authorization enforcement (gates T1, T2, T11):** Does the Go back-end independently enforce object-level and role authorization on every REST and WebSocket action, or does it trust JWT claims / client-supplied permission state? This is the single most load-bearing assumption in the model.
- **JWT signature verification (gates T1):** Does the back-end verify the JWT signature on every request, given the front-end decodes claims with `atob`/`JSON.parse` and never verifies?
- **CSRF posture (gates T6):** Are session cookies set with `SameSite=Strict/Lax` and `Secure`, and does the back-end require an anti-CSRF token or origin check on state-changing calls?
- **Deployment headers (gates T5):** Is a fronting proxy / CDN / WAF adding CSP, HSTS, and X-Frame-Options in production, compensating for `server.js` emitting none? If not, the app ships with zero header defenses.
- **`authorization_url` trust (gates T9):** Can the back-end ever be induced to return an attacker-controlled `authorization_url`, and is that URL constrained server-side?
- **Refresh-token rotation & absolute session lifetime (session hygiene):** Does the back-end rotate refresh tokens and enforce a maximum session lifetime? (SESSION_MANAGEMENT_ANALYSIS flagged MED-2/MED-3 as back-end-dependent.)
- **`everyone` group semantics (gates T1/T2):** Who can create an authorization entry with `provider_id == 'everyone'`, and does the back-end bound that grant?
- **Proxy deployment (gates T14):** Is `TMI_ENABLE_API_PROXY` ever enabled in production, and how is `TMI_PROXY_TARGET` constrained?

## 8. Recommended mitigations

| mitigation | threat_ids | closes_class | effort |
|---|---|---|---|
| Emit a strict CSP (`default-src 'self'`, no `unsafe-inline`/`unsafe-eval`) plus HSTS / X-Frame-Options / X-Content-Type-Options as real response headers from server.js or a fronting proxy; retire the dead client-side security-headers interceptor | T5,T4 | partial | M |
| Route every markdown render path through DOMPurify (including `renderer.link`) and scheme-allowlist href/src so `javascript:`/`data:` cannot reach the DOM; re-enable or layer Angular's sanitizer | T4 | yes | S |
| Treat all frontend authorization as UX-only and enforce object-level + role authorization server-side on every API/WS action; document this contract so the SPA's client checks are never the security boundary | T1,T2,T11 | yes | L |
| Pin and lock all dependencies including the server.js runtime deps (move them into the locked manifest), pin base images by digest, and replace EOL/unmaintained client libs (@antv/x6 2.x, pdf-lib, html2canvas) or sandbox their input | T7,T8 | partial | L |
| Validate every URL assigned to `window.location.href` (including backend-supplied `authorization_url`) against an origin + scheme allowlist before navigating | T9 | yes | S |
| Enforce anti-CSRF (`SameSite`/`Secure` cookie attributes + per-request CSRF token or origin check) on all state-changing API/WS calls | T6 | partial | M |
| Schema-validate and size-cap all inbound WebSocket frames and imported diagram JSON before they reach typed handlers; keep null-prototype parsing | T10 | partial | M |
| Run the production container as a non-root user across all Dockerfiles (adopt the `Dockerfile.oci` appuser pattern everywhere) | T13 | yes | S |
